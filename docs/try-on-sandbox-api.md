# Try-On Sandbox API Contract

This document records the current AI FitFabrica Try-On sandbox API before real generation adapters are introduced.

The sandbox is backend-owned. The web client uploads files, creates a job, polls status/result endpoints, and renders the response. The browser must not call AI providers, calculate credits, or decide workflow repair/retry rules.

## Scope

Included:

- Multipart job creation for one human photo and one garment photo.
- Typed job lifecycle statuses.
- Typed status polling.
- Typed result polling.
- Sandbox-only async lifecycle modes for frontend verification.
- Backend-selected storage adapters for job and upload persistence.
- Deterministic fake generation for completed jobs.
- Provider-runtime generation routing behind a backend flag.
- Sandbox cost event with no real credit charge.
- SQL-ready job aggregate shape that can persist on the portable PostgreSQL baseline.

Excluded:

- `sandbox_fake` remains the default.
- `provider_runtime` still persists a deterministic placeholder artifact.
- `vertex_virtual_try_on` now returns real provider image bytes through the backend-owned object storage path.
- A backend quality-verifier stage now runs after generation before result exposure.
- A backend repair contour now runs when quality verification recommends a local fix.
- No Vertex, Gemini, Imagen, or real AI generation.
- No real credits deduction.
- No marketplace search.
- No durable job history after backend restart in default in-memory mode.

## Storage Backend

The sandbox API remains stable while storage is selected server-side.

- Default local/test mode uses in-memory job storage and in-memory object storage.
- When portable SQL infrastructure is configured, Try-On jobs persist through the SQL runtime repository.
- `object_storage_backend=s3` enables S3-compatible upload persistence only when `object_storage_bucket_name` is configured.
- `try_on_generation_backend=sandbox_fake|provider_runtime` remains valid for the existing fake and provider-runtime contours.
- `try_on_generation_backend=sandbox_fake|provider_runtime|vertex_virtual_try_on` selects whether Try-On result generation stays on the local fake adapter, moves through the provider-runtime image-editing contour, or uses the dedicated Vertex Virtual Try-On backend.
- Stored upload references are internal backend state and are not exposed as public result URLs.
- The provider-runtime path still stores a deterministic placeholder artifact until a real provider byte-return path is connected.
- The `vertex_virtual_try_on` path requires `ENABLE_REAL_TRY_ON_GENERATION=true`, `VERTEX_PROJECT`, `OBJECT_STORAGE_BACKEND=s3`, `POSTGRES_DSN`, `OPERATIONS_QUEUE_BACKEND=redis`, and `REDIS_URL`.
- `TRY_ON_VERTEX_FAILURE_FALLBACK_BACKEND=none|provider_runtime|sandbox_fake` controls whether a real Vertex rollout is allowed to fall back to another backend when the primary generation call fails.
- Production-like environments must keep `TRY_ON_VERTEX_FAILURE_FALLBACK_BACKEND=none` unless `ALLOW_UNSAFE_TRY_ON_VERTEX_FALLBACK_IN_PRODUCTION=true` is set explicitly.
- All generation backends now pass through a backend-owned quality-verifier contour before a result can be returned as `completed`.
- When the verifier returns `repair_recommended`, the backend runs a repair pass and verifies the repaired artifact again before deciding `completed` vs `failed`.
- `try_on_quality_verifier_backend=deterministic|model_backed` selects whether verification stays purely deterministic or adds a structured provider-backed decision layer over backend facts.
- If the model-backed verifier provider is unavailable, the backend falls back to deterministic verification instead of breaking the Try-On job lifecycle.
- `try_on_repair_backend=deterministic|provider_runtime` selects whether repair stays on deterministic fallback logic or uses the provider-runtime image-editing contour for local correction.
- `try_on_stylist_backend=deterministic|model_backed` selects whether the final user-facing fit/style explanation stays deterministic or is generated by a separate structured reasoning step after quality and repair.

See `docs/try-on-durable-storage-activation.md` for the durable storage checklist and `docs/try-on-real-activation.md` for the real Vertex Try-On rollout guard, fallback policy, and rollback procedure.

## POST /api/try-on/jobs

Creates a backend-owned Try-On sandbox job.

Request type: `multipart/form-data`.

Required file fields:

- `human_photo`
- `garment_photo`

Accepted content types:

- `image/jpeg`
- `image/png`
- `image/webp`

Default max file size: `10 MB`, configurable through backend settings.

Optional sandbox-only form field:

- `sandbox_lifecycle_mode`

Allowed values:

- `complete` - default. The route creates an accepted job, dispatches worker execution, and the worker completes with a deterministic sandbox result.
- `pending` - the route creates an accepted job but does not dispatch worker execution; result polling returns `not_ready`.
- `failed` - the route creates an accepted job, dispatches worker execution, and the worker ends the job with `job_failed`.

Success response: `202`.

Response body:

```json
{
  "job_id": "try_on_<id>",
  "workflow_type": "try_on",
  "status": "accepted",
  "input_metadata": [],
  "status_url": "/api/jobs/try_on_<id>/status",
  "result_url": "/api/jobs/try_on_<id>/result"
}
```

The create response always confirms that the backend accepted the job. Final lifecycle state is observed through status/result polling.

## GET /api/jobs/{job_id}/status

Returns the current job status, full status history, and sandbox cost events.

Success response: `200`.

Response body:

```json
{
  "job_id": "try_on_<id>",
  "workflow_type": "try_on",
  "status": "generating",
  "status_history": [
    {
      "status": "accepted",
      "stage": "accepted",
      "message": "Job accepted.",
      "occurred_at": "2026-05-26T00:00:00Z"
    }
  ],
  "cost_events": [
    {
      "event_type": "try_on_sandbox_generation",
      "estimated_units": 0,
      "charge_status": "not_charged",
      "charged_credits": 0,
      "occurred_at": "2026-05-26T00:00:00Z"
    }
  ]
}
```

Possible statuses:

- `accepted`
- `generating`
- `quality_checking`
- `completed`
- `failed`

## GET /api/jobs/{job_id}/result

Returns the completed result or a typed lifecycle response.

Completed response: `200`.

```json
{
  "status": "completed",
  "job_id": "try_on_<id>",
  "workflow_type": "try_on",
  "result": {
    "job_id": "try_on_<id>",
    "workflow_type": "try_on",
    "result_image": {
      "kind": "sandbox_placeholder",
      "url": "/images/shared/try-on-sandbox-result.webp",
      "alt": "Sandbox Try-On result preview"
    },
    "quality_report": {
      "verdict": "pass",
      "confidence": 0.91,
      "checks": [],
      "limitations": []
    },
    "stylist_note": "Final stylist explanation generated by the backend-owned stylist step.",
    "input_metadata": [],
    "completed_at": "2026-05-26T00:00:00Z"
  }
}
```

Not-ready response: `202`.

```json
{
  "status": "not_ready",
  "job_id": "try_on_<id>",
  "workflow_type": "try_on",
  "current_status": "accepted",
  "status_url": "/api/jobs/try_on_<id>/status"
}
```

Failed response: `409`.

```json
{
  "error": {
    "code": "job_failed",
    "message": "Try-On job failed.",
    "details": {
      "job_id": "try_on_<id>"
    }
  }
}
```

## Error Envelope

All typed Try-On API errors use this shape:

```json
{
  "error": {
    "code": "missing_required_file",
    "message": "Human and garment photos are required.",
    "details": {}
  }
}
```

Known error codes:

- `missing_required_file`
- `unsupported_content_type`
- `empty_file`
- `file_too_large`
- `job_not_found`
- `result_not_ready`
- `job_failed`
- `storage_unavailable`

## Frontend Contract

The web client should:

- Create jobs through `POST /api/try-on/jobs`.
- Poll `GET /api/jobs/{job_id}/result`.
- If result polling returns `not_ready`, call `GET /api/jobs/{job_id}/status`.
- Continue polling until `completed` or `failed`.
- Show backend errors from the typed error envelope.
- Treat credits as display-only sandbox data.

The web client must not:

- Call AI/model providers directly.
- Store secrets.
- Calculate or deduct credits.
- Decide repair/retry workflow rules.
- Persist job state outside backend-owned APIs.
