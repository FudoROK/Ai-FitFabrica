Итоговая формула продукта
Любой зарегистрированный пользователь с credits может запускать базовые рабочие сценарии: примерка, поиск похожего, подбор образа и карточка
товара. Бизнес-профиль не является разрешением на карточку; он превращает разовую генерацию в рабочую систему магазина: стиль бренда, каналы
продаж, шаблоны, история товарных проектов и экспорт. Подключение магазина нужно только для внешних действий: публикация, импорт каталога,
синхронизация цены и остатков.

2. Уровни доступа и расширения
Маршрут: Workspace access model | Тип: Product logic
Backend управляет
Agents work
Уровень
Что открывает
Что требуется
Уровень 1: аккаунт + credits
Дает право запускать действия: примерка, похожее, образ, карточка товара, фото на
модели, описание, скачивание результата.
Ничего не нужно, кроме входа и credits.
Уровень 2: бизнес-профиль
Открывает настройки магазина: бренд, стиль, каналы продаж, шаблоны карточек,
бизнес-историю, экспорт под площадки.
Нужна кнопка «Зарегистрировать бизнес» и форма
/workspace/business-profile.
Уровень 3: подключенный магазин
Открывает публикацию, импорт каталога, синхронизацию товаров, цены и остатков
через API/фиды/разрешенные методы.
Нужны доступы магазина: API key, OAuth, токен,
кабинет продавца или ручной экспорт.
Не входит в первую версию
Сотрудники, роли команды, приглашения, права доступа внутри магазина.
Оставить как future: Team / Agency.
Правило UX
•
Не заставлять пользователя регистрировать бизнес до первой
карточки.
•
Сначала показать пользу: создай карточку, скачай результат.
•
После результата предложить бизнес-профиль для шаблонов, бренда
и экспорта.
•
При публикации на площадку показать «подключите магазин».
Правило backend
•
Backend проверяет capabilities, а не просто business/non-business.
•
Credits списываются за workflow и шаги.
•
Business profile добавляет context в Product Card Workflow.
•
Connected store добавляет external publish/import/sync actions.

Назначение
Экран запуска примерки: пользователь загружает фото человека и

Кнопки и действия

Фото человека -> загрузка JPEG/PNG/WebP
•

•
Сгенерировать примерку -> create try-on job

•

Business profile может подставлять требования к сохранению деталей
товара, если пользователь запускает примерку как продавец.
Credits
Списываем credits за workflow. Repair по нашей ошибке не должен
списывать повторно.

15. Профиль бизнеса
Маршрут: /workspace/business-profile | Тип: B2B profile
Backend управляет
Agents work
Назначение
Профиль магазина или бренда: описание, каналы продаж, логотип,
визуальный стиль, moodboard, фон и требования к сохранению
деталей товара.
Кнопки и действия
•
Сохранить изменения -> создать/обновить бизнес-профиль
•
Каналы продаж -> Instagram / сайт / WB / Ozon / Kaspi / manual export
•
Сгенерировать стиль по описанию -> AI-подсказка
•
Добавить moodboard -> референсы бренда
•
Слайдер точности деталей -> правила quality check
Бизнес-расширение
•
После сохранения бизнес-профиля открываются B2B-расширения:
шаблоны, бренд-контекст, бизнес-история, экспорт.
•
Кнопки публикации остаются заблокированы, пока не подключен
магазин/API.
•
Высокая точность деталей может увеличивать стоимость workflow.
Credits
Заполнение профиля не списывает credits. Credits списываются, когда
профиль используется в генерации/анализе.

22. Что backend должен хранить
Маршрут: Data model overview | Тип: Backend foundation
Backend управляет
Agents work
Сущность
Что хранит
users
Аккаунт, auth, настройки, общий credits balance.
user_style_profiles
Размеры, посадка, цвета, бюджет, B2C-предпочтения.
business_profiles
Магазин, описание, каналы продаж, стиль, moodboard, требования к контенту.
jobs
Тип workflow, input, status, progress, error, credits estimate/final.
product_cards
Товарные проекты, версии карточек, тексты, изображения, экспорт.
artifacts
Исходные фото, generated results, quality reports, repair versions, ZIP.
credit_ledger
Пополнения, списания, возвраты, внутренняя стоимость.
connected_stores
Площадка, статус подключения, scopes, last sync, errors.
marketplace_offers
Похожие товары, цены, competitor snapshots, source metadata.
Правило данных
Frontend не решает, что пользователь может делать. Он получает от backend список enabled_capabilities: product_card_create, business_templates,
manual_export, marketplace_publish, catalog_import, catalog_sync. Так мы не связываем интерфейс жестко с одним тарифом или одной ролью.

23. Capabilities вместо хаоса ролей
Маршрут: GET /workspace/state | Тип: API contract
Backend управляет
Agents work
{
  "user_id": "user_123",
  "credits_balance": 300,
  "has_business_profile": true,
  "business_profile_id": "biz_456",
  "connected_stores": ["manual_export"],
  "enabled_capabilities": [
    "try_on",
    "similar_search",
    "product_card_create",
    "business_templates",
    "content_pack",
    "manual_export"
  ],
  "locked_capabilities": [
    {
      "name": "marketplace_publish",
      "reason": "requires_store_connection"
    }
  ]
}
Почему так лучше
•
Нет жесткого деления «клиент / бизнес» в каждом компоненте.
•
Одна рабочая область умеет адаптироваться под состояние
пользователя.
•
Можно постепенно открывать функции без переделки интерфейса.
•
Кнопки получают понятные причины locked/disabled.
Как UI использует capabilities
•
Если есть product_card_create - показываем «Создать карточку».
•
Если есть business_templates - подставляем стиль магазина.
•
Если marketplace_publish locked - показываем кнопку, но с подсказкой
подключения.
•
Если не хватает credits - показываем цену и переход к пополнению.
Первый API для workspace
GET /api/workspace/state должен стать главным контрактом для frontend:
баланс, профиль, business status, enabled capabilities, последние jobs и
pending actions.

24. Короткое ТЗ на реализацию Workspace
Маршрут: Implementation checklist | Тип: For Codex / frontend
Backend управляет
Agents work
1. Общий Workspace
•
Оставить единый маршрут /workspace.
•
Sidebar строить по capabilities.
•
Не делать отдельные /business-workspace и /client-workspace.
•
Добавить CTA «Зарегистрировать бизнес» там, где профиля нет.
2. Product Card доступен всем
•
Не блокировать /workspace/product-card без business profile.
•
Блокировать только generation, если нет input или credits.
•
После результата предложить бизнес-профиль для шаблонов и
экспорта.
•
Публикацию блокировать до connected store.
3. Business Profile
•
Форма: название, описание, каналы, стиль, moodboard, фон, strictness
деталей.
•
Сохранение не списывает credits.
•
После сохранения включить business capabilities.
•
Без сотрудников и team roles.
4. States
•
empty, uploading, input_ready, queued, analyzing, generating,
quality_check, repair, ready, failed.
•
Недостаточно credits -> /workspace/credits.
•
Нужен бизнес -> /workspace/business-profile.
•
Нужен магазин -> /workspace/integrations.
Главный принцип для разработки
Каждый экран должен отвечать на четыре вопроса: что пользователь загрузил, что сейчас делает backend, сколько это стоит в credits, что
пользователь может сделать дальше. Всё остальное - украшения. Красивый интерфейс без этой логики будет опять «витриной».
